An inline loading indicator ‚Äî pulsing `dots` (default), a rotating `spinner`
arc, bouncing `typing` dots (chat "waiting for a reply"), a `morph`ing shape,
or a `bar` to place at the top of the content it covers. The bar is
indeterminate by default; give it a `value` (0‚Äì100) and it becomes a
determinate fill that completes, with progressbar semantics. Everything
follows the surrounding font size and text color (`currentColor`), so it works
standalone, inside buttons, or next to text.

For buttons, prefer the Button's own `loading` prop: it renders this spinner,
announces `aria-busy` and ignores presses while active.

### Import

```svelte
<script>
  import Loading from "@design-system/svelte/Loading.svelte";
</script>
```

### Live status ‚Äî a running description

For a process that reports its progress in steps, pass `status`: a message that
renders **inside** the polite `role="status"` region and is announced on every
change (the region is `aria-atomic`, so each new message is read in full). This
is how you surface a succession of backend-reported states ‚Äî `"Connecting‚Ä¶"` ‚Üí
`"Fetching records‚Ä¶"` ‚Üí `"Rendering‚Ä¶"` ‚Äî rather than a single static line.

```svelte
<!-- `status` is whatever the backend last reported; each change is announced. -->
<Loading variant="spinner" status={currentStep} />
```

It differs from `label` (a static accessible name) and from `showLabel` /
`detail` / the percentage (which are `aria-hidden` on purpose, so a fast-ticking
value isn't re-announced on every frame). Use `status` for discrete, meaningful
steps; use `detail` for a value that changes continuously. `status` is ignored
on a determinate `bar` (a progressbar announces its value via `aria-valuetext`).

### Overlay pattern

To mark a whole region as reloading, set `overlay` on the Loading and
`aria-busy="true"` on the region (which must be positioned, e.g.
`position: relative`). The indicator fills and centers over the region; by
default a translucent **veil** dims it and blocks pointer interaction while
busy. Set `veil={false}` to overlay just the indicator ‚Äî no dimming, no pointer
blocking ‚Äî for cases where a surrounding modal already guards interaction (this
is what `DropZone` uses while the native file dialog opens). For richer progress
(bytes, file counts, percentages) use
[Progress](/invisible-ui/components/progress/) instead.

```svelte
<div aria-busy={busy} style="position: relative">
  ‚Ä¶region content‚Ä¶
  {#if busy}<Loading overlay label="Reloading" />{/if}
</div>
```

### When to show a loader ‚Äî response-time UX

Whether to show a loader at all depends on how long the wait is (Nielsen's
response-time limits, and the Doherty threshold of ~400 ms):

| Wait | Perceived as | What to show |
| --- | --- | --- |
| **< 0.1 s** | Instant | Nothing ‚Äî a loader here only adds noise. |
| **0.1 ‚Äì 1 s** | Noticeable, still in flow | Usually nothing; avoid a layout that looks frozen. |
| **1 ‚Äì 10 s** | Attention wanders | An indeterminate indicator (`dots` / `spinner`) so the wait reads as "working". |
| **> 10 s** | Flow broken | A determinate `bar` with a percentage/`detail`, and ideally a way to cancel. |

**The delay rule ‚Äî never flash a loader.** Don't show the indicator the instant
an action starts. Reveal it only after a short delay (~**100‚Äì200 ms**): if the
work finishes first, the user never sees a spinner. A loader that appears and
vanishes within a frame reads as a glitch, not as progress. Once shown, keep it
visible for a brief minimum (~300‚Äì500 ms) so it can't blink out either.

This is built in ‚Äî pass `delay` (ms) and Loading stays hidden until it elapses.
Mount it as soon as the work starts and remove it when the work ends; if that
happens before `delay`, nothing is ever shown:

```svelte
{#if busy}<Loading variant="spinner" delay={150} />{/if}
```

The `DropZone` uses exactly this: the native file dialog can take up to ~1 s to
open, so its spinner (`delay={150}`) appears only if the dialog is slow ‚Äî
instant opens never flash one.

### Accessibility

- By default the indicator is a polite `role="status"` with an accessible name
  (`label`; when omitted it comes from the i18n catalog, so `LocaleProvider`
  localizes it). A determinate bar is a `progressbar` with `aria-value*`, and
  `detail` doubles as its `aria-valuetext`.
- The visible text is `aria-hidden`: the accessible name/value already carries
  it, and a ticking percentage inside a live region would be re-announced on
  every change.
- Set `decorative` when the surrounding region already announces the loading
  state (e.g. a button with its own label) ‚Äî the dots are then hidden from
  assistive tech.
- The pulse respects `prefers-reduced-motion`: the dots stay visible, static.
