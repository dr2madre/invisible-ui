A **quick search**: search by keyword and pick from a list, in a modal (the
pattern is also marketed as "command palette", but nothing here is a command).
It composes the **Dialog** (portal, focus trap, scroll lock, Escape / backdrop
close, focus restore) and the **Combobox** (search, filter, keyboard) — no new
core code. What it searches is up to the application: content, pages, files,
people, or the service's own actions and features.

Give items a **`group`** ("Pages", "Actions") to section the results by type
under a header, like a product-wide quick search: ungrouped results come first,
groups follow in order of first appearance, and a group whose results are all
filtered out disappears with its header. Keyboard traversal is flat — arrows
cross group boundaries in display order.

No keyboard shortcut is built in: bind `open` and wire the shortcut in the
application (the demo above uses <kbd>⌘S</kbd> / <kbd>Ctrl+S</kbd>).

Unlike the family's simple dialogs, it asks nothing and interrupts nothing —
it is a search tool inside the same modal shell.

### When to use which dialog

- **[Dialog](/invisible-ui/components/feedback/dialog/)** — generic modal for any content; the head of the family.
- **[Alert Dialog](/invisible-ui/components/feedback/dialog/alert-dialog/)** — an acknowledgement: a message with
  nothing to cancel (one button).
- **[Confirm Dialog](/invisible-ui/components/feedback/dialog/confirm-dialog/)** — verify or accept before proceeding:
  cancel stops, confirm proceeds (`urgent` to interrupt).
- **[Prompt Dialog](/invisible-ui/components/feedback/dialog/prompt-dialog/)** — asks for a single value
  (optional, `required`, or gated by `confirmValue`; `urgent` to interrupt).
- **Search Dialog** — a quick search: search by keyword and
  pick from a list.
- **[Sheet Dialog](/invisible-ui/components/feedback/dialog/sheet-dialog/)** — a task in an edge-anchored panel
  (`side` variant, optional drag-to-dismiss).

### Accessibility

- A modal `role="dialog"` (`aria-modal`) containing a `role="combobox"` search
  input over a `role="listbox"` of results. On open, focus moves to the input and
  is trapped; **Escape** / a backdrop press close it and restore focus.
- Typing filters; **Arrow Up/Down** move the highlight (`aria-activedescendant`),
  **Enter** selects the active result and closes.
- Grouped results sit in `role="group"` containers named via `aria-label`, so
  screen readers announce the section when entering it; the visible header is
  hidden from AT to avoid double reading.
- A visually-hidden `role="status"` region announces the filtered result count
  ("3 results available" / "No results found.") on every keystroke, so
  screen-reader users hear the effect of their typing.

### Example

```svelte
<script>
  let open = false;
  const items = [
    { value: "home", label: "Home", group: "Pages" },
    { value: "new", label: "New File", group: "Actions" },
    { value: "save", label: "Save", group: "Actions" },
  ];
</script>

<svelte:window
  on:keydown={(e) => {
    if ((e.metaKey || e.ctrlKey) && e.key === "s") {
      e.preventDefault();
      open = true;
    }
  }}
/>

<SearchDialog {items} bind:open onSelect={(value) => run(value)}>
  <span slot="trigger">Search…</span>
</SearchDialog>
```

Themeable via `--ds-search-dialog-*` (panel width, inset, radius) plus the
shared `--ds-dialog-*` overlay tokens.
