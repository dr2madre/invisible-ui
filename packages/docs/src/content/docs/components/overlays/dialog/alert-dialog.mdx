A modal acknowledgement — the accessible equivalent of `window.alert()`, the
first of the platform's *simple dialogs*. It interrupts to communicate an
important message (`role="alertdialog"`, announced immediately by screen
readers), but there is **nothing to cancel**: one button takes note and closes.
Escape and a backdrop press are equivalent to the button; set
`closeOnOutsideClick={false}` to require an explicit press (e.g. a notice the
user must actively acknowledge).

If the dialog offers a *choice* — something that can stop a process — it is not
an alert: use [Confirm Dialog](/invisible-ui/components/overlays/dialog/confirm-dialog/)
(with `urgent` if it must interrupt), or
[Prompt Dialog](/invisible-ui/components/overlays/dialog/prompt-dialog/) if it asks for
a value.

### When to use which dialog

- **[Dialog](/invisible-ui/components/overlays/dialog/)** — generic modal for any content; the head of the family.
- **Alert Dialog** — an acknowledgement: a message with nothing to cancel (one button).
- **[Confirm Dialog](/invisible-ui/components/overlays/dialog/confirm-dialog/)** — verify or accept before proceeding:
  cancel stops, confirm proceeds (`urgent` to interrupt).
- **[Prompt Dialog](/invisible-ui/components/overlays/dialog/prompt-dialog/)** — asks for a single value
  (optional, `required`, or gated by `confirmValue`; `urgent` to interrupt).

### Wording

The default label is the conventional "OK", but prefer naming the outcome in
context: **Done** after an operation, **Close** for a plain notice,
**I understood** when the user is taking on an obligation. Same principle as
the family's [button copy guideline](/invisible-ui/components/overlays/dialog/confirm-dialog/#wording):
buttons name outcomes, not gestures.

### Accessibility

- Panel is `role="alertdialog"` with `aria-modal="true"`, named by the title and
  described by the message (both required). Screen readers announce the content
  immediately — that is what the role is for.
- Focus starts on the only button; focus is trapped and body scroll locked;
  focus returns to the trigger on close.
- **Escape** and a backdrop press acknowledge exactly like the button
  (`onDismiss` runs in every case) — there is no destructive path to protect.

### Example

```svelte
<AlertDialog
  title="File deleted"
  description="“report-q3.pdf” was permanently deleted."
  dismissLabel="Done"
  onDismiss={() => refresh()}
>
  Show outcome
</AlertDialog>
```

Themeable via `--ds-dialog-*` (shared with Dialog).
